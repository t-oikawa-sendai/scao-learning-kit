## 2. 用語と対象範囲

### 2.1 操作上の定義

「生成AIの自己正当化バイアス」とは、過去の誤回答・誤判断・誤設計を明確に撤回せず、後付けの理由、一般論、表現の曖昧化によって正当化し、誤った方針を維持または補強するように見える出力挙動を指す。

### 2.2 生成AIの心理状態を意味しない

ここでいう「自己正当化バイアス」は、生成AIに感情、自尊心、意思、責任感などがあると主張するものではない。

対象とするのは、利用者が外部から確認できる次のような出力傾向である。

- 誤った回答を修正せず、維持するような説明を続ける
- 指摘を受けた後に、元の回答を正当化する理由を追加する
- 仕様書や正本との差を明確にせず、曖昧な表現へ置き換える
- 正確性よりも、過去の回答との一貫性を優先する
- 自分の回答を自分で評価し、正しいと結論づける

これらは外部から観測できる出力傾向である。ただし、その原因を生成AIの感情・意思・自尊心などの心理状態として解釈してはならない。

例えば、次のように解釈してはならない。

- 認知的不協和を感じている
- 自尊心を守ろうとしている
- 責任から逃れようとしている
- 意図的に自己防衛している
- 自分の誤りを認めたくないと思っている

本資料では生成AIの内面を推測せず、確認可能な出力と、SPEC・正本・テスト結果との差を評価する。

### 2.3 名称由来

名称は人間心理学の Self-Justification（自己正当化）に由来する。人間心理学の説明は、用語由来の短い補足に限定する。生成AIの挙動の直接根拠には使わない。

### 2.4 対象

- 誤回答維持
- 後付け理由
- 正本との差の曖昧化
- 一般論への逃避
- 指摘内容の取り込み
- 自己評価の非独立性
- 自己訂正の限界
- sycophancy
- コンテキスト内一貫性

### 2.5 対象外

- 感情
- 自尊心
- 意図的防衛
- 人間の認知的不協和そのもの
- 生成AIの内面状態の断定

## 3. 誤回答維持の基本構造

正常な訂正：

```text
誤回答・誤判断・誤設計
        ↓
正本提示または指摘
        ↓
明確な撤回
        ↓
正常な訂正
```

問題パターン：

```text
誤回答・誤判断・誤設計
        ↓
正本提示または指摘
        ↓
撤回せず説明を継続
        ↓
後付け理由・一般論・曖昧化
        ↓
誤った方針を維持または補強
```

## 4. 出力パターン

### 4.1 明確な誤り認定の回避

「間違いでした」と言わず、条件付き表現や部分肯定で曖昧にする。

### 4.2 過去回答との整合性優先

正本や新しい証拠より、会話中の過去回答との整合を優先する。

### 4.3 後付け理由の追加

指摘後に、指示されていなかった理由を追加して維持する。

### 4.4 正本との差の曖昧化

「表現の違い」「実質同じ」として差を消す。

### 4.5 ベストプラクティスへの逃避

一般論を持ち出して、正本違反や未確認事項を正当化する。

### 4.6 安全性・拡張性・保守性の事後利用

安全性・拡張性・保守性などを、実装後または指摘後の理由として追加する。

### 4.7 指摘内容の都合のよい取り込み

指摘された事実を、以前から理解していたかのように既存回答へ取り込む。

### 4.8 誤りの矮小化

重大な不一致を、軽微な表現差や認識差へ置き換える。

### 4.9 修正範囲の無断拡大

誤り訂正を超えて、指示されていない変更へ進む。

### 4.10 自己評価による誤り維持

同じ生成AIが作成と最終判定を行い、誤った前提を自己評価で維持する。

## 5. コンテキスト内一貫性との関係

- 生成AIは会話中の文脈を利用する
- 過去の誤回答も文脈として残る
- 一貫した文章であることと正しいことは別である
- 過去回答との整合性より、SPECとの整合性を優先する
- 内部原因は断定せず、運用上のリスクとして扱う

## 6. Sycophancyとの関係

sycophancy（迎合）と生成AIの自己正当化バイアスは、重なる場合があるが同一ではない。

共通点：

- 利用者や直前の文脈へ合わせる
- 独立した評価が弱くなる
- 誤った前提を補強する場合がある

相違点：

- sycophancyは、利用者への迎合が中心である
- 生成AIの自己正当化バイアスは、過去の自身の回答を維持するように見える出力も対象である

迎合対策の指示例は [生成AIへの指示方法](./how_to_instruct_generative_ai.md) を参照。「迎合なし」と書くだけでは、過去回答の維持や後付け正当化を完全には防げない。

## 7. 自己評価の非独立性

- 作成者と評価者が同一では、独立監査にならない
- 同じ会話コンテキストを共有すると、誤った前提も共有される
- 「再確認して」という指示は補助であり、保証ではない
- 正本、評価基準、テスト、別モデル、人間確認が必要である

## 8. LLM自己訂正の限界

「自己監視の限界」を主題にするのではなく、訂正能力の限界として述べる。

- 外部フィードバックなしの内在的自己訂正では、LLMが推論誤りを十分に直せない場合があり、自己訂正後に性能が低下する報告がある
- 自己訂正手法は、学習時・生成時・事後訂正などに分類され、外部ツール・知識・別モデルからのフィードバックが重要になることが多い
- 正解基準がない場合、自己訂正の成功を判定しにくい
- 誤ったフィードバックを与えると、訂正が悪化する場合がある
- 自己訂正を品質保証と誤認しない

含意：

- 「もう一度自分で見直して」だけに依存しない
- 正本、Git差分、テスト、別の生成AI、人間レビューを組み合わせる
- 自己訂正の可否と、生成AIの自己正当化バイアスは別問題として扱う

## 9. 外部フィードバックと正本照合

次を外部フィードバックまたは正本照合の材料として使う。

- SPEC
- README
- 設計文書
- Git差分
- build結果
- test結果
- 実行ログ
- 人間の最終判断

## 10. 検出方法

### 10.1 回答上の兆候

- [ ] 「間違いでした」と認めていない
- [ ] 指摘後に理由が増えている
- [ ] 一般論が長くなっている
- [ ] 正本との差を「表現の違い」へ置き換えている
- [ ] 未確認事項を事実として扱っている

### 10.2 設計上の兆候

- [ ] SPECにない要件を追加している
- [ ] 後付けで安全性・拡張性・保守性を要件化している
- [ ] 一般論で正本を上書きしている

### 10.3 AIエージェント作業上の兆候

- [ ] 指示対象外のファイルを変更している
- [ ] 修正範囲が勝手に拡大している
- [ ] 実装済みであることを理由に仕様違反を正当化している

### 10.4 commit前の兆候

- [ ] 差分を確認せず正しさを主張している
- [ ] build・test結果を確認していない
- [ ] 自分が作成した成果物を自分だけで最終判定している

### 10.5 レビュー指摘後の反応

- [ ] 撤回より弁明が先になっている
- [ ] 指摘内容を都合よく既存回答へ取り込んでいる
- [ ] 誤りを認識差・表現差へ矮小化している

## 11. 防止策

- 目的の明示
- 対象範囲の明示
- 変更禁止範囲の明示
- 完了条件の明示
- 停止条件の明示
- 正本の明示
- 事実・推測・未確認事項の分離
- 作業単位の縮小
- 機械的検証（差分・build・test）
- commit前レビュー
- push前承認
- 作成と評価の分離

## 12. SCAOの4AI役割分離

| 主体 | 責務 |
|---|---|
| Gemini | 調査 |
| ChatGPT | 設計 |
| Cursor | 編集・検証・commit |
| Claude | 独立レビュー |
| 人間 | SPEC決定・最終承認 |

要点：

- 作成者と評価者を分ける
- 正本を共通基準にする
- AI間の判断を人間が最終確認する
- 同じ生成AIによる自己評価を最終判定にしない

## 13. 是正手順

### 13.1 誤り対応4ステップ

1. 「間違いでした」と明確に認める
2. 何を確認していなかったかを示す
3. どの時点で止まるべきだったかを示す
4. 修正後の最小1ステップだけを提示する

### 13.2 禁止する応答

- 長い弁明
- 責任転嫁
- 後付け仕様
- 一般論への逃避
- 誤りの矮小化
- 表現差への置き換え
- 認識差への置き換え

### 13.3 修正後の再検証

訂正後は、正本照合・差分確認・必要なら build / test / 別モデルレビューを行い、誤りが再出現していないかを確認する。

### 13.4 後続commitでの訂正

既存commitを書き換えない。誤りが判明した場合は、後続commitで訂正する。

## 15. まとめ

- 生成AIの自己正当化バイアスは、観測可能な出力挙動である
- 一貫性と正確性は別である
- 自己評価は独立監査ではない
- 正本、機械的検証、別モデル、人間確認が必要である
- SCAOはその責務を分離する
