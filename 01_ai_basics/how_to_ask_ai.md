## 1. Purpose（目的）

AIへの依頼文を、意図が正確に伝わる形に整理できるようにする。
依頼文の基本要素をそろえることで、意図とずれた回答を減らす。

## 2. Target Audience（対象読者）

- AIへの頼み方が定まっていない初学者
- 依頼しても意図とずれた回答が返ってきて困っている人

## 3. Learning Goals（学習目標）

- 悪い依頼と良い依頼の違いを説明できる。
- 依頼文の基本要素をそろえて書ける。
- 簡易版として一部を省略してよい場面を判断できる。

## 4. Main Content（本文）

### 4.1 悪い依頼例

```
アプリ作って
```

- 何を作るのか分からない。
- 何のためか分からない。
- どんな条件があるのか分からない。

このような依頼だと、AIは推測で答えるため、意図とずれた回答になりやすい。

### 4.2 依頼文の基本要素

依頼文は、次の要素をそろえると意図が伝わりやすい。

1. 目的：何のためにやるのか。
2. 背景・前提：どういう状況か、分かっている条件は何か。
3. 対象範囲：どこまでを対象にするか。
4. 変更禁止範囲：触ってほしくない部分、してほしくないこと。
5. 完了条件：何を満たせば「できた」と言えるか。
6. 検証方法：結果をどう確認するか。
7. 停止条件：どうなったら止まって確認してほしいか。
8. 出力形式：どんな形（Markdown、表、箇条書きなど）で答えてほしいか。

### 4.3 良い依頼例

```
# 目的
家計簿アプリのREADMEの下書きを作りたい。IT職業訓練校のポートフォリオ提出用。

# 背景・前提
- 個人利用の小規模アプリ
- 使用技術は未定

# 対象範囲
READMEの下書きのみ。実装コードは今回は不要。

# 変更禁止範囲
- 実在しないライブラリ名を使わない
- 勝手に機能を増やさない

# 完了条件
- 概要・目的・使い方の見出しが含まれている
- 日本語で書かれている

# 検証方法
見出しの抜けがないか、こちらで目視確認する。

# 停止条件
使用技術を決める必要が出たら、いったん確認してほしい。

# 出力形式
Markdownの見出し付きで出力してください。
```

### 4.4 各要素の書き方のコツ

- 目的：最初に書く。方向性が定まる。
- 背景・前提：分からないことは「未定」と書いてよい。
- 対象範囲：やることとやらないことを分ける。
- 変更禁止範囲：「実在しないライブラリを使わない」「勝手に増やさない」など。
- 完了条件：確認できる形（見出しがある、日本語である等）で書く。
- 検証方法：どうやって正しさを確かめるかを書く。
- 停止条件：判断に迷う場面で止めてほしい条件を書く。
- 出力形式：例を1つ示すと、より正確に伝わる。

### 4.5 簡易版では一部を省略してよい

すべての依頼で全要素を書く必要はない。
小さな質問や軽い依頼では、目的と出力形式だけでも十分なことが多い。

- 簡単な調べ物：目的＋出力形式
- 少し複雑な依頼：目的＋前提＋制約＋完了条件
- 重要・大きな依頼：全要素をそろえる

依頼が大きく・重要になるほど、要素をそろえる価値が高い。

### 4.6 迎合なし・反論歓迎の指定例

AIは、頼み方によっては安易に肯定してしまうことがある。
批判的な視点がほしいときは、次のように依頼するとよい。

```
この設計案について、迎合せずに指摘してください。
問題点・抜け漏れ・リスクがあれば、遠慮なく反論してください。
「良いと思います」だけの回答は不要です。
```

なお、これは攻撃的な口調を求めるものではない。また、「迎合なし」と書くだけでは、誤った過去回答の維持や後付け正当化を完全には防げない。詳しくは [AIの自己正当化バイアス 基礎](./self_justification_bias_basic.md) を参照。

## 5. Practical Checkpoints（実務上の確認点）

- 目的が依頼文の先頭に書かれているか。
- やること・やらないこと（対象範囲・変更禁止範囲）が分かるか。
- 完了条件と検証方法が確認できる形になっているか。
- 依頼の重要度に対して、要素の粒度が適切か（簡易版で十分か、全要素が必要か）。
