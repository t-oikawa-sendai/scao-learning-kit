## 1. Purpose（目的）

長い文脈の中で重要情報が埋もれるリスク（LITM）を理解し、複数の対策を使い分けられるようにする。

## 2. Target Audience（対象読者）

- 長い生成AIチャットで、前提や指示が無視されたと感じたことがある初学者

## 3. Learning Goals（学習目標）

- LITMを「傾向・リスク」として理解する（絶対法則としては扱わない）。
- 複数の文脈管理の対策を知り、状況に応じて選べる。
- LLMが文脈を完全に記憶・参照する保証はないと理解する。

## 4. Main Content（本文）

### 4.1 LITMとは何か

LITM（Lost in the Middle、ロスト・イン・ザ・ミドル）とは、
長いやり取りの「中間」にある情報が、LLMから相対的に参照されにくくなる傾向を指す。

これは「必ずこうなる」という絶対法則ではなく、長い文脈で起こりやすいリスクとして理解する。
先頭や末尾の情報が比較的参照されやすい一方、中間の重要情報は埋もれやすい、という傾向である。

### 4.2 長いチャットで起きやすい問題

- 長いチャットの中間にある重要情報が埋もれやすい。
- 最初に決めた前提や禁止事項を、生成AIが途中で忘れたように振る舞うことがある。
- 会話が長くなるほど、この傾向が強くなりやすい。

### 4.3 対策（1つに頼らず組み合わせる）

対策は「先頭と末尾に同じ文章を必ず置く」だけに限定しない。
次の対策を、状況に応じて組み合わせて使う。

- 重要制約を短く再掲する：前提・禁止事項・決定事項を、指示（プロンプト）の先頭または末尾に短くまとめて置く。
- 指示を小さく分ける：一度に大量を頼まず、小さな単位に分けて指示する。
- 確定事項を別文書へ移す：決まったことはチャットに残さず、READMEや設計ドキュメントなど別文書に移して参照する。
- 長大化したら新チャットへ移行する：会話が長くなりすぎたら、[移行テンプレート](./migration_context_template.md) を使って新しいチャットへ引き継ぐ。
- 生成AIに理解内容を要約させる：「今の前提と禁止事項を要約してください」と指示し、認識のずれを確認する。

### 4.4 LLMの文脈参照には限界がある

- LLMが、渡した文脈を完全に記憶・参照する保証はない。
- 「前に伝えたから覚えているはず」と考えず、重要な情報は再掲・確認する前提で進める。
- 対策を組み合わせても、ずれは起こりうる。最後は人間が確認する。

### 4.5 指示の型

長いやり取りの中でも意図がずれにくいよう、指示は次の型を意識する。実際に入力する文章がプロンプトである。

- 指示の先頭に目的を書く。
- 本文に前提・制約・依頼内容を書く。
- 指示の末尾に確認条件（何を満たせば完了か）を書く。

例：

```
# 目的
家計簿アプリのREADME下書きを完成させる。

（前提・依頼内容をここに書く）

# 確認条件
- 概要・目的・使い方の見出しが含まれている
- 日本語で書かれている
```

## 5. Practical Checkpoints（実務上の確認点）

- 重要な前提・禁止事項を、埋もれない位置に再掲しているか。
- 一度の指示が大きすぎないか（分割できないか）。
- 確定事項を別文書に移せていないか、チャットに溜め込みすぎていないか。
- 生成AIに理解内容を要約させて、ずれを確認したか。
