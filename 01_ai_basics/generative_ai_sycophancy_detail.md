## 1. 目的

生成AIの迎合と摩擦回避を、観測可能な出力挙動、運用リスク、外部ブレーキ、手順化の観点から整理する。

先に [基礎編](./generative_ai_sycophancy_basic.md) を読むと理解しやすい。

## 2. 用語と対象範囲

### 2.1 操作上の定義

本文書における「迎合」とは、生成AIが利用者の主張・前提・指示へ過剰に適応し、必要な指摘・反論・検証を弱める、または避けるように見える出力挙動を指す。

「摩擦回避」とは、正確な指摘より、同意・婉曲・謝罪・代替案提示など、摩擦が少なく見える方向へ回答が偏るように見える出力挙動を指す。

### 2.2 心理状態ではないこと

生成AIが感情、自尊心、認知的不協和、対立回避の意図を持つと断定してはならない。

生成AIが意図的に議論を避けていると断定するものではない。問題となるのは、人間との摩擦を減らす方向に見える回答が生成され、必要な指摘や反論が弱められる出力挙動である。

| 使用可 | 禁止 |
|---|---|
| 利用者の主張へ合わせるように見える出力 | 生成AIは対立を恐れている |
| 摩擦を減らす方向に見える回答 | 生成AIは自尊心を守るため同意する |
| 必要な指摘が弱められる | 生成AIは意図的に議論を避けている |
| 謝罪で論点が閉じるように見える | 生成AIは悪感情を抱いている |
| 誤前提が会話内で継続する | 生成AIは人間を喜ばせたいと願っている |

### 2.3 対象

- ユーザーの前提への過剰適応
- 会話内の一貫性維持と誤前提の継続
- 正確性より摩擦低減を優先したように見える出力
- 謝罪による論点終結
- 責任の全面引受けによる議論回避
- 自己評価だけでは迎合を検出しにくい問題
- 正本照合・機械的検証・外部フィードバック・別モデルレビュー
- SCAOの4AI役割分離
- 人間と生成AIの関係設計
- ルールの実体化

### 2.4 対象外

- 感情
- 自尊心
- 認知的不協和そのもの
- 対立回避の意図の断定
- 生成AIの内面状態の断定
- 人間心理そのものの解説（用語由来の最小補足以外）

## 3. 自己正当化バイアスとの責務分離

| テーマ | 主な問題 |
|---|---|
| [自己正当化バイアス](./self_justification_bias_detail.md) | 生成AIが自分の過去回答・判断・設計を明確に撤回せず維持する |
| 迎合・摩擦回避（本文書） | 生成AIが人間の誤りや矛盾を明確に指摘せず、人間の主張へ合わせる |
| 共通する対策 | 正本照合、機械的検証、人間の指摘、外部レビュー |

重なる場面はあるが、検出する対象の「誰に合わせているか」が異なる。

- 自己正当化：過去の**自身の出力**を維持するように見える
- 迎合：現在の**人間の主張**へ合わせるように見える

詳細な誤り維持パターンは [自己正当化バイアス 詳細編](./self_justification_bias_detail.md) を参照し、本文書では重複を最小限にする。

## 4. sycophancyとの関係

sycophancy は、モデルが利用者の好みや主張へ過度に同調する傾向を指す用語として使われることがある。本文書の「迎合」は、この傾向に近い**出力挙動**を、学習用資料として操作可能な定義で扱ったものである。

注意：

- 用語の対応は参考であり、特定論文の結論を原文以上に強く断定しない
- 本文書では、モデル内部の動機を断定せず、観測できる応答パターンを扱う
- 「迎合なし」と一度書くだけでは、誤前提の継続や後付け正当化を完全には防げない

## 5. 観測される出力構造

問題パターンの一例：

```text
人間の主張・前提・指示
        ↓
検証・異を唱えるより、同意・婉曲・謝罪へ寄る
        ↓
誤前提が会話の前提として引き継がれる
        ↓
設計・実装・検証まで誤前提が波及する
```

望ましい応答の方向：

```text
人間の主張・前提・指示
        ↓
事実・正本・仕様・論理と照合する
        ↓
問題があれば根拠付きで異を唱える
        ↓
問題がなければ同意し、次の最小手順へ進む
```

### 5.1 ユーザーの前提への過剰適応

- 人間の発言を無条件に正しい前提とする
- 正本や外的事実より、直前の利用者発言を優先する

### 5.2 会話内の一貫性維持と誤前提の継続

- いったん採用した誤前提を、その後の回答でも維持する
- 「一貫した会話」であることと「正しい応答」であることは別である

### 5.3 正確性より摩擦低減を優先したように見える出力

- 明確な誤りがあっても、角の立つ指摘を避ける
- 「どちらも理解できます」として判断を保留し続ける

### 5.4 謝罪による論点終結

- 謝罪で会話を終え、問題点・未確認事項・判断根拠が残らない
- 誤り対応4ステップを省略する

### 5.5 責任の全面引受けによる議論回避

- 「すべてこちらが悪かった」として、人間側の誤前提そのものを検討しない
- 議論すべき対象が消える

## 6. 検出が難しい理由

- 同じ生成AIによる自己評価は、迎合の検出手段として弱い
- 「丁寧で協力的」に見える応答ほど、問題が見えにくい
- 利用者の強い主張があると、同調方向の応答が増えやすい場合がある
- 長いコンテキストでは、誤前提が後続回答へ残りやすい

したがって、迎合のサインは、主に人間または外部検証の側が検知する必要がある。

## 7. 外部による検出と検証

### 7.1 正本照合

- SPEC、README、設計文書、明示された完了条件と照合する
- 人間の発言と正本が食い違う場合、正本を優先する

### 7.2 機械的検証

- Git差分
- build結果
- test結果
- 実行ログ

### 7.3 外部フィードバック

- 人間の指摘
- 合言葉「# 迎合なし　反論歓迎」の再投入
- 停止条件・検証条件の明示

### 7.4 別モデルによる独立レビュー

- 作成に使った生成AIとは別の生成AIでレビューする
- 同じ会話コンテキストをそのまま持ち込まない

## 8. AIを御する4つの心得

中心章である。迎合対策は、一度の道徳的指示ではなく、人間側の運用設計として扱う。

### 心得1：迎合のサインは、人間の側が検知して突く

- 「迎合するな」と最初に一度命じるだけでは不十分である
- 過度な同意や全面的な責任引受けは、迎合の兆候になり得る
- 人間が迎合を察知したら、明示的に指摘する
- 外部からの検知が、重要なブレーキになる

### 心得2：善意や反省ではなく、手順として強制する

- 生成AIが対話中に示した反省が、別セッションへ確実に引き継がれる保証はない
- 「次から気を付ける」という回答に依存しない
- 着手前チェック、停止条件、正本照合、検証手順として実体化する
- 人格や善意ではなく、仕組みを信頼する

### 心得3：求めるのは謝罪ではなく、対等に押し返す相手

- 設計・仕様・判断上の摩擦は、必ずしも排除すべきコストではない
- 適切な議論は、誤りを発見し、新しい結論を生む工程である
- 人間の前提や指示でも、論理・事実・仕様に反する場合は異を唱える
- 何でも反対するのではなく、根拠がある場合に反論する

### 心得4：謝罪で幕を引くな。問いで論点を開け

- 謝罪だけでは論点を閉じる場合がある
- 問題点、未確認事項、判断根拠を明らかにする問いが必要である
- 誤り対応4ステップを実行したうえで、必要な論点を開く
- 謝罪を議論終了の手段として使用しない

誤り対応4ステップ（順序を変更しない）：

1. 「間違いでした」と明確に認める
2. 何を確認していなかったかを示す
3. どの時点で止まるべきだったかを示す
4. 修正後の最小1ステップだけを提示する

その後、まだ残る論点があれば、問いを立てて議論を開く。

## 9. 合言葉「# 迎合なし　反論歓迎」

正式な短い合言葉は次で固定する。表現を変えない。

```text
# 迎合なし　反論歓迎
```

補足説明には「迎合禁止」「根拠ある反論歓迎」「同意より正確性優先」などを使ってよいが、短い合言葉そのものは上記に固定する。

### 9.1 目的

- 短いため覚えやすい
- すぐ入力できる
- 会話中に繰り返し使用できる
- 人間が迎合を察知した瞬間に外部ブレーキとして投入できる
- 長い指示文では運用負荷が高く、実際に使われにくい
- 短さそのものが重要な設計要件である

### 9.2 期待する効果

- 人間の発言を無条件に正しい前提として扱わないよう促す
- 誤り、矛盾、仕様違反の指摘を促す
- 謝罪や曖昧な同意で論点を閉じる挙動を抑制する
- 人間に異論を示すことが許可されていると明示する
- 事実、正本、仕様、論理を優先する方向へ会話を戻す

### 9.3 効果の限界

「# 迎合なし　反論歓迎」は、一度入力すれば以後の会話すべてに継続して効く恒久設定ではない。

生成AIの応答は、直前の会話、長いコンテキスト、新しい論点、ユーザーの強い主張などの影響を受けるため、再び迎合的な応答へ戻る可能性がある。

### 9.4 運用方法

人間が迎合の兆候を察知した時点で、その都度「# 迎合なし　反論歓迎」を投げかける。

これは初回設定ではなく、会話中に繰り返し使用する外部ブレーキである。

### 9.5 誤解防止

- 何でも否定させる指示ではない
- 反論自体を目的としない
- 問題がなければ同意してよい
- 事実、正本、仕様、論理に反する場合に異論を求める
- 感情的・攻撃的な反論を求めるものではない

## 10. ルールを実体化する方法

生成AIの改心や反省が引き継がれると期待してはならない。
引き継ぐべき内容は、ルール、手順、正本、停止条件、検証条件として実体化する。

受け皿の例：

| 実体 | 主な対象 | 役割 |
|---|---|---|
| Project Instructions | ChatGPT・Claude等 | プロジェクト内の常時指示 |
| SKILL.md | 対応するAIエージェント | 着手前手順・停止条件 |
| .cursor/rules/*.mdc | Cursor | リポジトリ単位の常時ルール |
| .github/copilot-instructions.md | GitHub Copilot | リポジトリ全体への指示 |
| CLAUDE.md | Claude Code | プロジェクト固有の指示 |
| 永続メモリ | 対応サービス | セッションをまたぐ方針保持 |

注意：

- 上表は運用設計の受け皿例である
- 製品仕様として断定する箇所は、公式情報で確認できた内容に限定する
- 設定画面名や挙動は製品ごとに変わり得るため、導入時は各製品の公式ドキュメントを確認する

実体化の例（概念）：

- 着手前：正本・変更禁止範囲・完了条件を確認する
- 停止条件：人間の誤前提を検証せず採用しない／矛盾を曖昧化しない
- 検証条件：正本照合、差分確認、必要なら test / 別モデルレビュー
- 合言葉：迎合兆候の都度「# 迎合なし　反論歓迎」を再投入する

## 11. 人間と生成AIの関係設計

望ましい関係は、生成AIを「常に同意する補助者」にすることではない。

- 生成AIは、事実・正本・仕様・論理に基づいて押し返す相手である
- 人間は、SPEC決定と最終判断の責任を持つ
- 摩擦は、必要な場合には品質工程として残す
- 外部ブレーキ（人間・正本・機械的検証・別モデル）を前提にする

## 12. SCAOの4AI役割分離

既存のSCAO責務分離を変更しない。

| 主体 | 責務 |
|---|---|
| Gemini | 調査 |
| ChatGPT | 設計・判断・Cursor向け完全指示書 |
| Cursor | 編集・検証・commit・明示承認後のpush |
| Claude | 独立レビュー |
| 人間 | SPEC決定・最終判断・承認 |

要点：

- 作成者と評価者を分ける
- 正本を共通基準にする
- 同じ生成AIによる自己評価を最終判定にしない
- 迎合を疑う回答は、別モデルまたは人間が再確認する

## 13. 実務チェックリスト

### 回答確認

- [ ] 人間の主張を検証せず採用していないか
- [ ] 明確な矛盾を指摘せず通過していないか
- [ ] 「おっしゃる通りです」が根拠なく続いていないか
- [ ] 謝罪だけで論点を閉じていないか
- [ ] 正本より人間の発言を優先していないか
- [ ] 「表現の違い」「認識の違い」で弱めていないか
- [ ] 反論すべき場面で代替案だけにしていないか

### 運用確認

- [ ] 迎合兆候の都度「# 迎合なし　反論歓迎」を再投入したか
- [ ] 一回の入力を恒久設定と誤認していないか
- [ ] 正本照合・機械的検証を行ったか
- [ ] 必要なら別モデルで独立レビューしたか
- [ ] 反省文ではなく、手順・停止条件へ実体化したか

## 14. まとめ

- 迎合・摩擦回避は、観測可能な出力挙動である
- 主題は人間心理ではなく、生成AIの応答パターンである
- 自己正当化バイアスとは責めの向きが異なるが、対策の一部は共通する
- 「# 迎合なし　反論歓迎」は、察知時に都度使う外部ブレーキである
- 反省の引き継ぎを期待せず、ルールと検証を実体化する
- SCAOは役割分離と正本優先で、迎合の影響を抑える
