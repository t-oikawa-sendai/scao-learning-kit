## 1. Purpose（目的）

READMEの役割を理解し、プロジェクトの入口として過不足なく書けるようにする。

## 2. Target Audience（対象読者）

- 初めてREADMEを書く初学者
- ポートフォリオの見せ方を整えたい人

## 3. Learning Goals（学習目標）

- READMEはプロジェクトの入口であると理解する。
- READMEに書くべき最低限の内容を挙げられる。
- 詳細を設計文書へ分け、READMEに詰め込みすぎないようにできる。

## 4. Main Content（本文）

### 4.1 READMEの役割

README（リードミー）は、プロジェクトの入口となる説明文である。
初めてそのプロジェクトを見た人が、「何なのか」「どう使うのか」を最初に理解するための場所である。

ポートフォリオでは、READMEが第一印象を決める。
中身がよくても、READMEが分かりにくいと評価されにくい。

### 4.2 READMEはプロジェクトの入口

- 最初に読まれることを前提に書く。
- 専門知識のない人でも概要が分かるようにする。
- 詳しい話より、まず「何ができるか」を伝える。

### 4.3 READMEに書くべき最低限の内容

- 何を作ったか（プロジェクトの概要）
- なぜ作ったか（作成の目的・背景）
- 何ができるか（主な機能）
- 使い方（動かし方、操作の流れ）
- 画面イメージ（スクリーンショットや図）
- 設計文書への導線（要求・要件・簡易設計へのリンク）
- 注意事項（制約、既知の問題など）

### 4.4 READMEに詰め込みすぎない

READMEの役割は、次に限定する。

- プロジェクトの入口
- 目的
- 概要
- 利用方法
- 設計文書への導線
- 現在の状態

要求・要件・詳細設計のすべてをREADMEに詰め込まない。
細かい仕様や設計の詳細は、別の設計文書に分け、READMEからはリンクで参照する。

### 4.5 テンプレートに含まれる項目

[README_TEMPLATE.md](./templates/README_TEMPLATE.md) には、上記に加えて次の項目がある。各項目は概要のみを短く書き、詳細は要求・要件・簡易設計へ分ける。

- 課題・解決・効果の概要（Problem / Solution / Benefit Summary）：現在の課題・開発目的・解決方針・システム機能・期待効果・完了判定基準を1〜2文ずつ。
- 用語集（Glossary）：プロジェクト固有の用語・略語。
- 管理者・レビュアー（Owners and Reviewers）：誰が責任を持ち、誰が確認するか。
- 画面概要（Screen Overview）：代表画面のみ。画面を持たない場合は「対象外。理由：画面を持たないため。」と書ける。

## 5. Practical Checkpoints（実務上の確認点）

- 初めて見る人が、概要・目的・使い方を理解できるか。
- 要求・要件・設計の詳細をREADMEに書きすぎていないか。
- 設計文書への導線（リンク）があるか。
- 現在の状態（開発中・完成など）が分かるか。
