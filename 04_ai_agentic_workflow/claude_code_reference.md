## 2. Claude Codeとは何か

Claude Code は、Anthropic が提供する、開発作業向けのAIエージェント製品である。一般的なチャット回答にとどまらず、リポジトリ内の調査、ファイル編集、コマンド実行などを伴う作業を支援する用途で位置付けられる。

正式表記は **Claude Code** である。

## 3. Claudeチャット画面との違い

| 観点 | Claude（チャット） | Claude Code（概念上の差） |
|---|---|---|
| 主界面 | 会話画面 | ターミナル／開発環境中心 |
| 主な成果 | 文章・提案・レビューコメント | リポジトリ操作を伴う作業支援 |
| ファイル変更 | 原則として直接は行わない（貼り付け・説明が中心） | 技術的にはファイル編集が可能 |
| コマンド実行 | 原則として環境外 | 技術的にはコマンド実行が可能 |
| SCAO上の典型用途 | 独立レビューなどの読取り・評価 | 機能があることと、SCAOでどの責務を与えるかは別問題 |

チャット版Claudeが「読む・考える・指摘する」ことに向くのに対し、Claude Code は「リポジトリ上で動く」能力を持つ。ただし、能力があることと、SCAOで実装担当に割り当てることとは同一ではない。

## 4. ターミナル型AIエージェントとしての位置付け

AIエージェントとは、生成AIがツール・ファイル・コマンド・外部サービスを利用し、複数工程を実行する場合の呼び方である。Claude Code は、その中でもターミナル／開発リポジトリを主戦場とするタイプとして理解するとよい。

学習上のポイント：

- 質問応答の延長ではなく、作業ループ（調査→変更→確認）を持つ
- 便利さより、承認・差分確認・停止条件が重要になる
- Cursor など他のエージェント型ツールと役割が重なり得るため、SCAOでは責務を人間が決める

## 5. リポジトリでの基本的な使い方（概念）

典型的な利用イメージは次である。

1. 対象リポジトリの作業ディレクトリで起動する
2. 目的・対象範囲・変更禁止範囲・完了条件を指示する
3. 調査結果と変更案を確認する
4. 実際のファイル差分とコマンド結果を確認する
5. 人間が承認した範囲だけを残す／進める

具体的な起動コマンド・認証手順・UI操作は製品更新で変わり得るため、本資料では記載しない。公式ドキュメントを確認すること。

## 6. 調査・計画・編集・検証の流れ

安全な使い方の骨格は、次の4段階に分けて考える。

```text
調査 → 計画 → 編集 → 検証
```

| 段階 | 期待する出力 | 人間が見る観点 |
|---|---|---|
| 調査 | 関連ファイル・現状の整理 | 正本・既存実装を読んでいるか |
| 計画 | 変更方針・対象範囲 | 余計な範囲へ広がっていないか |
| 編集 | ファイル変更 | 指示対象だけか |
| 検証 | 差分・テスト・確認結果 | 未確認事項を事実化していないか |

大きな作業を一度に任せない。段階ごとに人間確認を挟む。

## 7. ファイル変更とコマンド実行の確認

Claude Code は、技術的にはファイル編集やコマンド実行が可能である。だからこそ、次を人間が確認する。

- どのファイルが追加・変更・削除されたか
- コマンドは調査用か、破壊的操作か
- 秘密情報（APIキー、パスワード、トークン、秘密鍵、個人情報）を扱っていないか
- 指示されていないリファクタや範囲拡大がないか

「うまく動いた」だけでは完了にしない。変更内容そのものを見る。

## 8. Git差分とテスト

推奨する確認：

- `git status` / `git diff` で意図した差分だけかを見る
- 必要なら関連テストを実行し、結果を確認する
- build / lint / test の失敗を「問題なし」と曖昧化しない
- commit や push は、SCAO運用では人間の承認条件に従う

AIエージェントの自己評価だけで「テストも問題ありません」と結論づけない。ログと差分を見る。

## 9. CLAUDE.mdの役割（概念）

多くのプロジェクトでは、リポジトリ固有の方針を文書として置き、AIエージェントに読ませる運用がある。Claude Code 向けには、`CLAUDE.md` をプロジェクト固有の指示置き場として使う考え方がある。

概念上の役割：

- プロジェクト固有の前提・禁止事項・完了条件を書く
- 「次から気を付ける」反省文に頼らず、手順として残す
- セッションをまたいで再利用しやすい指示の受け皿にする

`CLAUDE.md` の書式・読込条件・優先順位は製品仕様として変わり得る。実際の挙動は公式ドキュメントで確認すること。迎合対策や停止条件を実体化する受け皿の一例としては、[生成AIの迎合と摩擦回避 詳細編](../01_ai_basics/generative_ai_sycophancy_detail.md) も参照。

## 10. 権限管理と危険な操作

概念としての安全運用：

- 必要最小限の権限だけを与える
- 破壊的コマンド（強制push、履歴改変、一括削除など）を安易に許可しない
- 本番データ・本番秘密情報へ触らせない
- 承認なしの remote push をデフォルトにしない
- 「便利だから全部許可」を避ける

権限モード名や設定手順は製品ごとに更新される。断定しない。導入時に公式の権限・安全ガイドを確認すること。

## 11. SCAOで使用する場合の役割境界

既存のSCAO責務分離を変更しない。

| 主体 | SCAO上の中心責務 |
|---|---|
| Gemini | 外部情報調査 |
| ChatGPT | 設計・判断・Cursor向け完全指示書 |
| Cursor | 編集・検証・commit・明示承認後のpush |
| Claude | 独立レビュー |
| 人間 | SPEC決定・最終判断・承認 |

重要：

- Claude Code は技術的にはファイル編集やコマンド実行が可能である
- ただし、SCAOでどの責務を与えるかは人間が決定する
- 現在のSCAO標準では、Claude の中心責務は独立レビューである
- Claude Code の機能と、SCAO上の責務を混同しない
- Claude が自動的に実装担当へ変更された、と読んではならない

つまり、「Claude Code が使える＝SCAOの実装担当がClaudeになった」ではない。実装を Cursor に置き、Claude（および必要なら Claude Code）をレビュー側に置く、といった運用選択は人間が決める。

## 13. まとめ

- Claude Code は、ターミナル／リポジトリ中心のAIエージェントとして理解する
- チャット版Claudeとの最大の差は、編集・実行を伴い得る点である
- それでも SCAO では、Claude の中心責務は独立レビューである
- 機能の有無と、運用上の責務割り当てを混同しない
- 差分・テスト・権限・人間承認を前提に使う
- 変動する手順は公式ドキュメントで確認する
